# Publish-targets schema + Step 7 sub-step generation (NEW in v5)

> **Purpose**: abstract the publish-side of Step 7 so the same skill drives PyPI + GHCR (e.g. an OSS Python toolkit like Evidentia) or Vercel + npm (a Next.js web app + companion mobile package) or cargo + Homebrew-tap (a Rust CLI) without per-project skill edits. v4 hardcoded PyPI/GHCR/GitHub-Release in `step-7-post-tag.md`; v5 reads `publish-targets.yaml` and generates the right sub-steps per declared target.

Driven by `.local/pre-release-review/publish-targets.yaml`. Generated by [first-run-bootstrap.md](first-run-bootstrap.md) at wizard time + edited by the operator thereafter.

## Schema

```yaml
# .local/pre-release-review/publish-targets.yaml
schema_version: 1   # bumped on breaking changes; skill rejects mismatched

targets:
  - id: <slug>              # unique per file; used in per-run JSON refs
    kind: <one of 9 + custom>
    enabled: true|false     # disable without deleting (e.g., during incident)
    repo: <project URL>     # github.com/<owner>/<repo> or equivalent
    packages: [<list>]      # for multi-package projects (PyPI, npm, cargo)
    verify: [<step IDs>]    # which Step 7 sub-step verifiers to run
    config: {<kind-specific>}  # extra config per kind (e.g., npm registry URL)
```

## The 9 pre-stubbed `kind:` values

### `pypi-trusted-publisher`

OIDC-published Python wheels with PEP 740 attestations.

```yaml
- id: evidentia-pypi
  kind: pypi-trusted-publisher
  repo: github.com/Polycentric-Labs/evidentia
  packages:
    - evidentia
    - evidentia-core
    - evidentia-ai
    - evidentia-api
    - evidentia-collectors
    - evidentia-integrations
    - evidentia-mcp
  verify:
    - pypi_artifact_hash_diff   # 7.2
    - pep740_attestation         # 7.3
    - slsa_provenance_v1         # 7.4 (optional; skip if not emitted)
    - published_sbom_osv_scan    # 7.6
    - fresh_venv_install_smoke   # 7.8
```

### `ghcr-container`

GitHub Container Registry images with cosign signing.

```yaml
- id: evidentia-container
  kind: ghcr-container
  repo: ghcr.io/polycentric-labs/evidentia
  verify:
    - cosign_keyless_verify      # 7.5a
    - docker_run_smoke           # 7.5b
    - container_cve_rescan       # 7.5c (Grype default per v5.0.1; Trivy opt-in)
  config:
    cosign:
      certificate_identity_regexp: "https://github.com/Polycentric-Labs/evidentia/.+"
      certificate_oidc_issuer: "https://token.actions.githubusercontent.com"
    docker_smoke_command: "evidentia version"
    expected_version_in_output: "v{VERSION}"
```

### `github-release`

GitHub Releases page with attached artifacts.

```yaml
- id: evidentia-github-release
  kind: github-release
  repo: github.com/Polycentric-Labs/evidentia
  artifacts:
    - evidentia-sbom.cdx.json
    - "*.sigstore.json"
  verify:
    - release_notes_audit          # 7.9
    - sbom_body_diff               # 7.6 (cross-checked with pypi-trusted-publisher)
    - artifacts_attached           # 7.9 sub-check
```

### `npm`

npm registry packages (covers JSR if you point `registry:`).

```yaml
- id: acme-webapp-mobile
  kind: npm
  repo: github.com/<owner>/<repo>
  packages:
    - "@acme/mobile"
  verify:
    - npm_version_published        # `npm view <pkg>@<ver>` returns
    - npm_integrity_sha512_match   # tarball hash matches build
    - npm_fresh_install_smoke      # `npm install -g <pkg>@<ver>` succeeds
  config:
    registry: https://registry.npmjs.org
```

### `vercel`

Vercel deployments (web apps).

```yaml
- id: acme-webapp
  kind: vercel
  repo: github.com/<owner>/<repo>
  verify:
    - vercel_deployment_id_matches_commit_sha
    - vercel_custom_domain_alias_intact   # app.example.com → deployment
    - vercel_health_endpoint_200          # GET /api/health
    - vercel_env_vars_present             # listed in config.env_vars below
  config:
    project_id: prj_xxx
    custom_domains:
      - app.example.com
    health_endpoint: /api/health
    env_vars:
      - <THIRD_PARTY_API_KEY>
      - <AUTH_PROVIDER_SECRET>
```

### `cloudflare-pages`

Cloudflare Pages or Workers deployments.

```yaml
- id: docs-site
  kind: cloudflare-pages
  repo: github.com/<owner>/<repo>
  verify:
    - cf_deployment_hash_matches_build
    - cf_route_mapping_intact
    - cf_custom_domain_resolves
  config:
    project_name: docs
    custom_domains:
      - docs.example.com
```

### `cargo`

Rust crates published to crates.io.

```yaml
- id: my-rust-cli
  kind: cargo
  repo: github.com/<owner>/<repo>
  packages: [my-rust-cli]
  verify:
    - cargo_info_published        # `cargo info <crate>` shows new version
    - cargo_checksum_diff         # local Cargo.lock vs published
    - cargo_fresh_install_smoke   # `cargo install <crate>` succeeds
```

### `homebrew-tap`

CLI distribution via Homebrew tap.

```yaml
- id: my-cli-tap
  kind: homebrew-tap
  repo: github.com/<owner>/homebrew-tap
  packages: [my-cli]
  verify:
    - tap_formula_version_bumped  # Formula/<cli>.rb version field updated
    - tap_sha256_matches_tarball  # SHA-256 in formula matches GH-release asset
    - tap_install_smoke            # `brew install <owner>/tap/<cli>` succeeds
```

### `docker-hub-container`

Docker Hub container images (alternative to GHCR).

```yaml
- id: my-image-dockerhub
  kind: docker-hub-container
  repo: docker.io/<owner>/<image>
  verify:
    - dockerhub_manifest_sha256_diff
    - cosign_keyless_verify
    - docker_run_smoke
```

### `custom` — escape hatch

For publish targets the 9 stubbed kinds don't cover. Caller provides `verify:` step IDs that resolve to skill-registered handlers (or, if no handler exists, the skill surfaces "verifier X not implemented; provide a shell command" prompt).

```yaml
- id: my-deno-deploy
  kind: custom
  repo: github.com/<owner>/<repo>
  verify:
    - deno_deploy_health_endpoint
  config:
    deno_deploy_project_id: xxx
    verify_handler_module: scripts/post_release/verify_deno_deploy.py
```

## Step 7 sub-step generation

At Step 7 entry, the skill loads `publish-targets.yaml`, validates the schema, and generates the Step 7 sub-step list dynamically:

```
Step 7 sub-steps for v0.10.4 (generated from publish-targets.yaml):

7.0 Wait for release.yml completion
7.1 [pypi-trusted-publisher] pypi_artifact_hash_diff for 7 packages
7.2 [pypi-trusted-publisher] pep740_attestation for 7 packages
7.3 [pypi-trusted-publisher] slsa_provenance_v1 for 7 packages
7.4 [ghcr-container] cosign_keyless_verify
7.5 [ghcr-container] docker_run_smoke (evidentia version → v0.10.4)
7.6 [github-release] release_notes_audit
7.7 [github-release] sbom_body_diff
7.8 [github-release] artifacts_attached
7.9 [pypi-trusted-publisher] published_sbom_osv_scan
7.10 [pypi-trusted-publisher] fresh_venv_install_smoke
7.11 [global] Scorecard re-run + score-regression check
7.12 [global] Memory + audit-log update + auto-generate security-review-vX.Y.Z.md (Q9)

Total: 12 sub-steps. Estimated time: 30-90 minutes.
```

Sub-step IDs map to handler modules in the skill's `references/post_release/` directory (one Python module per `verify:` step ID).

## Schema validation

The skill validates `publish-targets.yaml` at:

1. **First-run-bootstrap write time** — strict schema check + dry-run of each `verify:` step ID to confirm handler exists
2. **Step 1 entry of every run** — re-validate (operator may have edited)
3. **Step 7 entry** — final validation before sub-step generation

Validation rules:

- `schema_version` matches the skill's supported version (currently `1`)
- Each `id` is unique within the file
- Each `kind` is one of the 9 stubbed values OR `custom`
- For `custom` kind, every `verify:` step ID has a corresponding handler module
- Each `repo` is a valid URL or `<host>/<owner>/<repo>` shape
- `config.cosign.certificate_identity_regexp` (when present) is a valid regex
- All `*_command` strings interpolate `{VERSION}` if they reference a version

## Handler module contract

Each `verify:` step ID resolves to a Python module under `references/post_release/<id>.py` (or per-project equivalent under `.local/pre-release-review/handlers/`). Signature:

```python
def verify(target: PublishTarget, version: str, context: VerifyContext) -> VerifyResult:
    """Run the verifier; return PASS/FAIL/SKIP + structured findings."""
```

Built-in handlers ship with the skill. Custom handlers can be added per-project under `.local/pre-release-review/handlers/`.

If validation fails, the skill refuses to advance Step 1 until fixed (analogous to first-run-bootstrap refusing to advance until skeletons land).

## Migration from v4

Existing v4 projects (Evidentia) have hardcoded PyPI+GHCR+GitHub-Release in `step-7-post-tag.md`. Migration:

1. Skill detects project has no `publish-targets.yaml` BUT has a tag history
2. Offers to auto-generate from project-shape detection + a `.github/workflows/release.yml` scan
3. Operator confirms the generated YAML
4. File written; future Step 7 runs use it

This makes v5 backward-compatible with v4 projects — no breaking change for Evidentia.
