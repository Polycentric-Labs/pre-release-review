# `/pre-release-review` v5.1 — a worked example

> **What this is.** A tutorial-style walkthrough of one full run of the
> `/pre-release-review` skill against a real project (Evidentia v0.10.3 →
> v0.10.4). Captures every STOP gate, every operator decision, every
> finding, and every skill-iteration observation. The raw session
> transcript (~507 KB) was archived locally during this run via the
> `/export-claude-session` skill but is intentionally NOT shipped here
> — this curated walkthrough is the high-signal artifact; the raw
> transcript adds wall-of-text without proportional information value
> for skill consumers.
>
> **Why this exists.** Solo-developer reviewers + auditors asked "does
> this skill actually run end-to-end against a real project, or is it
> aspirational?" This is the answer.
>
> **Skill version**: `2026.05.24-v5.1` (first ship under v5.1; v0.10.0
> → v0.10.3 ran under v4). **Result**: 17th consecutive PROCEED-CLEAN
> on the v0.7.x → v0.10.x line.

---

## The project at a glance

**Evidentia** is an open-source GRC engine — Python `uv`-workspace,
7 packages, ships to PyPI Trusted Publisher (PEP 740 attestations) +
GHCR (cosign-signed; SLSA Provenance v1) + GitHub Releases (CycloneDX
SBOM attached). 268 source files, ~3,370 tests, mypy `--strict` clean.

**This cycle (v0.10.4)** ships the OCSF symmetry loop close
(`evidentia gap analyze --format ocsf` is the symmetric counterpart to
v0.10.0 SARIF emit + v0.10.1 OCSF ingest) + a 13th MCP tool wrapping
`verify_ar_file` + v0.10.3 `/code-review` polish + the new CHANGELOG-
presence CI gate (lesson from a v0.10.3 move-tag re-fire incident).

10 commits in `v0.10.3..HEAD`, 2642 LOC delta, 1 new source file.

## Step 1 — Process review (~10 minutes)

**Goal**: align on scope, capabilities, threat-model freshness, and
publishing-authority posture before any parallel research kicks off.

### 1.0 Project-shape detection (NEW in v5)

The skill runs auto-detection at Step 1 entry. For Evidentia it
identifies:

- **Primary shape**: `python` (driven by `pyproject.toml` + `uv.lock`)
- **Build system**: `uv`
- **Test runner**: `pytest`
- **Publish targets**: `pypi-trusted-publisher`, `ghcr-container`,
  `github-release` (from `.github/workflows/release.yml` + `Dockerfile`
  + workflow patterns)
- **Protected branches**: `main` is protected with 6 required status
  checks but `enforce_admins: false`

The shape JSON is persisted to `.local/pre-release-review/project-shape.json`
so subsequent runs reuse it silently.

### 1.5.2 Per-run JSON freshness check (NEW v5 per #4 / Guideline #12)

The skill scans `.local/pre-release-review/runs/` for in-flight reviews
within the 4-hour ceiling. Latest prior JSON is ~24h old with
`completed_at: None` (abandoned earlier session). Not resumable; the
skill creates a fresh JSON: `2026-05-24T21-41-48Z-v0.10.4-prototype.json`.

This JSON is the **Guideline #12 push-gate token** — any subsequent
push command requires either a fresh JSON (< 4h) OR a verbatim bypass
phrase (`PUSH TO MAIN BYPASS AUTHORIZED` / `STALE REVIEW ACCEPTED — <reason>`
/ `DOC FRESHNESS BYPASS — <reason>`).

**Operator chose option 2 — full re-read** even though v0.10.4 is a
patch. Rationale logged: "prototype run for skill v5.1; maximize
